## 1. 現状評価サマリー

### 良い点

移行の完成度は高い。以下の点は安定しており、変更の必要がない。

- **@layer components の徹底**: 全コンポーネント・プロジェクトファイルが `@layer components` で囲まれており、Tailwind utility との優先度関係が正しく成立している
- **CSS 変数基盤の整備**: 色・gutter・header-height・form 系が `:root` / `.p-form` スコープで定義済み。Tailwind config も CSS 変数を参照しており、値の一元管理ができている
- **ビルドパイプラインの設計**: SCSS → concat → PostCSS の 3 段構成は、SCSS 関数を先に解決してから Tailwind に渡すという要件に対して正しい設計。ビルド時間も約 1.2 秒と実用的
- **namespace 統一**: 全ファイルが `@use "../global" as g;` で共通 API にアクセスする一貫したパターン
- **readme.md の完成度**: 現況ドキュメントとして十分な情報量。新規参入者でも構成を追える

### 注意が必要な点

以下は「壊れている」わけではないが、品質・保守性・Tailwind 活用の観点で改善余地がある。

## 3. A: コード品質の改善（低リスク）

### A-1: ハードコード色の CSS 変数化

**現状**: Phase 4 で SCSS 変数 → CSS 変数の移行を行った際、`color.scale()` で計算していた派生色がハードコード化された。以下のパターンが散在している。

```scss
// 現状（複数ファイルに散在）
border-left-color: #67c45d;  // was color.scale(g.$clr1, $lightness: 14%)
color: #888888;              // was color.scale(g.$clrg600, $lightness: 0%)
background-color: #aad94d;   // was color.scale(g.$clr2, $lightness: 15%)
color: #a9840a;              // was color.scale(g.$clr3, $lightness: 15%)
border-color: #47a73c;       // was color.scale(g.$clr1, $lightness: -5%)
background-color: #4bb140;   // was color.scale(g.$clr1, $lightness: -5%) (icon)
```

**対象ファイル**:
- `component/_post.scss`: `#67c45d` × 2
- `component/_style.scss`: `#888888`
- `project/_footer.scss`: `#aad94d`
- `component/_typ.scss`: `#a9840a`
- `project/_form.scss`: `#47a73c`, `#4bb140`
- `component/_button.scss`: `#4fba43` (= `var(--clr1)` と同値だが直書き)

**提案**: 使用頻度の高い派生色を `:root` に CSS 変数として追加する。

```scss
// _tailwind-base-layer.scss :root に追加
--clr1-light: #67c45d;   // clr1 の明るめ派生
--clr2-light: #aad94d;   // clr2 の明るめ派生
--clr3-light: #a9840a;   // clr3 の明るめ派生
--clr1-dark: #47a73c;    // clr1 の暗め派生（≒ --clr1-hover に近い）
```

**判断ポイント**:
- 1 箇所しか使われない色（`#4bb140` 等）は CSS 変数にせずそのまま残してよい
- `--clr1-hover: #3f9536` はすでに定義済み。`#47a73c` との関係を確認し、統合可能か判断する（値が近いため同じ意味の可能性あり）
- `#888888` は `var(--clrg600)` とほぼ同値（`--clrg600: #858585`）。意図的な差かどうか確認が必要

**リスク**: 低。CSS 変数の追加と参照の差し替えのみ。出力 CSS の値は変わらない。

### A-2: デッドコードの除去

**現状**: 複数ファイルに合計 200 行以上のコメントアウト済みコードが残存している。

**主な箇所**:

| ファイル | 行数 | 内容 |
|----------|------|------|
| `component/_style.scss` L11-161 | ~151 行 | 旧リストパターン（全体がコメントアウト） |

| `layout/_header.scss` L76-91 | ~15 行 | コメントアウトされたメディアクエリ |

**提案**: これらのコメントアウト済みコードを削除する。必要になった場合は git 履歴から復元できる。

**リスク**: 低。ビルドに影響しないコメントの削除のみ。

### A-3: `_login.scss` のブレークポイント不統一

**現状**: `component/_login.scss` 内で `@media (min-width: 576px)` がハードコードされている。他の全ファイルは `@media #{g.$sm}` を使用。

```scss
// 現状（_login.scss）
@media (min-width: 576px) { ... }

// 他の全ファイル
@media #{g.$sm} { ... }
```

**提案**: `@media #{g.$sm}` に統一する。

**リスク**: 低。値は同じ（576px）なので出力 CSS に変化なし。

### A-4: `_typ.scss` の `rem4` 修正

**現状**: `component/_typ.scss` の L313 付近に `rem4` という未定義の参照がある（`g.rem(4)` の誤記と推測）。

**提案**: 該当箇所を確認し、`g.rem(4)` に修正する。ビルドがエラーにならず素通りしている場合、出力値が意図と異なる可能性がある。

**リスク**: 低。ただし実際の表示への影響は要確認。

**優先度が低い理由**: 現状動作しており、分割しなくても保守できるレベル。無理に分割すると `@use` 依存が増える。

## 4. B: Tailwind ネイティブへの接近（中リスク）

### B-1: `$space_values` の二重管理解消

**現状**: spacing トークンが SCSS（`global/_variables.scss` の `$space_values`）と Tailwind（`tailwind.config.js` の `theme.spacing`）の両方で定義されている。値は同じだが、追加・変更時に両方を更新する必要がある。

```
SCSS:  $space_values: (0_5: 4, 0_75: 6, 1: 8, ...)
JS:    spacing: { 0.5: '4px', 0.75: '6px', 1: '8px', ... }
```

**現在の `$space_values` 使用状況を確認する必要がある**:
- `$space_values` が SCSS 内のループやマップアクセスで参照されているか
- 参照がなければ SCSS 側を削除し、Tailwind config のみに一本化できる

**提案**:
1. まず `$space_values` の参照箇所を精査する
2. SCSS 内で直接参照がなければ（Phase 2 で utility 生成が Tailwind に移行済みのため）、`$space_values` を削除して Tailwind 一本化
3. 直接参照がある場合は現状維持（二重管理のコメントを追記）

**リスク**: 中。削除前に全参照箇所の確認が必須。

---

### B-2: gutter 系 CSS 変数の整理可能性

**現状**: `--unit`, `--space`, `--gutter`, `--gutter-row` が `_tailwind-base-layer.scss` の `:root` で 6 ブレークポイント分定義されている。`get_vw()` を使った vw 値が各ブレークポイントで再計算されている。

**確認すべき点**:
- `--unit` と `--space` の実際の使用箇所がどの程度あるか
- `--gutter` と `--gutter-row` は Tailwind config の `extend.padding` / `extend.gap` 経由でも参照されている（`px-gutter-row` 等）
- vw ベースの値が本当に必要か、固定 px/rem に置き換え可能か

**提案**: 即座の変更は不推奨。ただし以下の情報を整理しておくと将来判断しやすい。
- `--unit` の参照箇所一覧（typography の `calc(var(--unit) * N)` パターン）
- `--space` の参照箇所一覧
- これらが vw である必然性（ビューポート幅に応じた流動的サイズが必要か、`clamp()` で代替可能か）

**リスク**: 高（変更する場合）。vw ベースのレスポンシブサイズが設計の根幹にあるため、安易な変更は表示崩れを招く。

---

### B-3: `theme()` 関数の活用拡大

**現状**: `theme()` は `body { font-family }` でのみ使用されている。

```scss
font-family: #{"theme(fontFamily.sans)"};
```

**Tailwind の `theme()` は PostCSS 段階で解決されるため、`@layer components` 内の値で `tailwind.config.js` に定義済みのトークンを直接参照できる**。

**活用可能な候補**:

| 現状 | theme() 化後 |
|------|------------|
| `border-radius: 6px` | `border-radius: #{"theme(borderRadius.DEFAULT)"}` |
| `transition: all 0.2s ease-in-out` | `transition: all #{"theme(transitionDuration.DEFAULT)"} ease-in-out` |
| `z-index: 1000` | `z-index: #{"theme(zIndex.header)"}` |

**提案**: `border-radius` と `z-index` については `theme()` 化の価値がある。`tailwind.config.js` を single source of truth にできるため。

ただし、以下の理由で段階的に行うべき:
- `#{"theme(...)"}` の記法は SCSS 内では直感的でない
- 全置換すると SCSS 側の可読性が下がる
- 新規コードや修正時に順次適用する程度でよい

**リスク**: 低。PostCSS 段階で値に展開されるため、出力 CSS は同一。

## 5. C: ビルド・運用の改善（低リスク）

### C-1: backup ファイルの整理

**現状**: `css/` ディレクトリに以下のファイルが残存している。

| ファイル | サイズ | 用途 |
|----------|--------|------|
| `css/style.css.bak` | 184 KB | Phase 5 開始時のバックアップ |
| `css/style.css.snapshot` | 132 KB | 用途不明 |
| `css/style.css.before-gutter` | 182 KB | gutter mixin 廃止前のバックアップ |

**提案**:
1. `.gitignore` に `css/*.bak`, `css/*.snapshot`, `css/*.before-*` を追加
2. 既存ファイルを削除（git 履歴で差分確認は可能）

**リスク**: 低。本番コードに影響なし。

---

### C-2: stylelint 設定の整備

**現状**: `package.json` に `lint:css` スクリプトと stylelint 関連パッケージが存在するが、`.stylelintrc` 設定ファイルが見当たらない。

**提案**: 最低限の設定ファイルを作成し、`@layer` 内での SCSS 記法を許可するルールを設定する。Tailwind との共存で必要なルール除外（`at-rule-no-unknown` 等）を含める。

**リスク**: 低。lint はビルドに影響しない。

---

### C-3: PHP watcher の最適化

**現状**: `watch:templates` が PHP ファイルの変更時に `css:build` 全体を再実行している。PHP 変更は Tailwind の utility クラススキャンに影響するが、SCSS コンパイルは不要。

**提案**: PHP 変更時は `css:postcss` のみ再実行する（SCSS コンパイルと concat をスキップ）。

```json
"watch:templates": "onchange './**/*.php' -e 'node_modules/**' -- npm run css:postcss"
```

**リスク**: 低。ただし、concat 済みの `css/style.css` が存在する前提。初回ビルド後であれば問題ない。

**効果**: PHP 変更時のリビルド時間が約 1.2 秒 → 約 0.8 秒に短縮（体感差は小さい）。

## 6. D: 長期的な検討事項（参考）

以下は今すぐ対応する必要はないが、将来の方向性として記録しておく。

### D-1: `@extend` の段階的解消

**現状**: `layout/_content.scss`, `component/_footer.scss`, `component/_typ.scss`, `project/_entrystep.scss` 等で `%placeholder` + `@extend` パターンが使用されている。

**Tailwind との関係**: `@extend` は `@layer` 内で動作するため現状問題ないが、Sass 依存度を高める要因。将来的に SCSS を廃止する場合の障壁になる。

**今やらない理由**: 動作しており、既存の BEM クラス構造と密接に結合している。解消するには PHP テンプレート側の変更も必要で、影響範囲が大きい。

---

### D-2: FontAwesome 5 → 6 移行

**現状**: FontAwesome 5.14.0 を使用。フォントファイルを直接バンドルしている。

**将来の選択肢**:
- FontAwesome 6 への更新（アイコン名の変更対応が必要）
- SVG アイコンへの移行（フォントファイル不要、パフォーマンス向上）
- Heroicons 等の Tailwind エコシステムのアイコンセットへの移行

**今やらない理由**: 現状動作しており、アイコンの使用箇所が SCSS mixin（`g.fa5-far()`, `g.fa-icon`）と密接に結合している。移行コストが高い。

---

### D-3: Sass 関数の CSS ネイティブ関数への段階置換

**現状**: `g.rem()`, `g.get_vw()`, `g.get_lh()` 等の Sass 関数がコンパイル時に値を計算している。

**CSS ネイティブでの代替可能性**:
- `g.rem(24)` → `1.5rem`（ビルド時計算 → 静的値）：すでに出力は同じ
- `g.get_vw(16, 960)` → CSS の `calc()` や `clamp()` で動的に実現可能

**今やらない理由**: Sass 関数はビルド時に解決されるため、ランタイムコストがない。CSS ネイティブ関数に置き換えるメリットは「SCSS への依存を減らす」のみで、パフォーマンス面の利点はない。

## 7. 対応しなくてよいもの

以下は調査した結果、変更不要と判断した項目。

### breakpoints の二重定義（SCSS + Tailwind）

readme.md に記載のとおり、用途が異なる（SCSS は `@media #{g.$md}` でクラス内展開、Tailwind は `md:` で PHP テンプレート）。統一するには全コンポーネント SCSS をリライトする必要があり、現状の共存設計が正しい。

### `@layer components` 外のベンダー CSS

FontAwesome の `@font-face` は `@layer` 外が正しい配置。Swiper / Micromodal / scroll-hint の `@keyframes` も `@layer` 外で正しく配置されている。

### `tailwind.config.js` の v3 形式

v4 では CSS-first config が推奨されるが、v3 形式の JS config も `@config` ディレクティブ経由で完全にサポートされている。現状の JS config は機能しており、移行の必然性がない。

### `important` 付きの Tailwind import

`@import "tailwindcss" important;` は意図的な設計。`@layer components` 内のスタイルが Tailwind utility で上書き可能であることを保証している。

### ビルドパイプラインの 3 段構成

concat ステップを PostCSS に統合する案もあるが、現状のパイプラインは明確で保守しやすい。最適化しても 100ms 程度の短縮にしかならず、複雑さが増すデメリットの方が大きい。
