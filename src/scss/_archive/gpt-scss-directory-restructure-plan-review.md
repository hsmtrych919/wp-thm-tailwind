# SCSS Directory Restructure Plan Review

対象: scss-directory-restructure-plan.md

前提:
- このレビューは、計画書に書かれている目的と方針を前提として読む
- つまり、FLOCSS の残骸を整理し、接頭辞を外し、Next.js / CSS Modules への接続性を上げる方向性自体は本レビューでは否定しない
- 今回の確認対象は、計画の構成、タスク分解、順序、完了条件の置き方
- 利用箇所の総当たり調査や、個々のクラスの実在確認までは行っていない

---

## 採用決定

### 1. Step 2 から Step 7 までが「ファイル移動・統合フェーズ」なのに、途中の完了条件がビルド確認に寄りすぎている

重要度: High

計画上、Step 2 から Step 7 はまだ接頭辞除去の前段であり、主に次をやるフェーズになっている。

- vendor の切り出し
- layout / component / project の物理統合
- style.scss の import 更新

この段階で本当に確認したいのは、単にビルドが通ることだけではない。
少なくとも次が成立している必要がある。

- build graph から落ちたファイルがないこと
- 統合元ファイルの内容が統合先へ移りきっていること
- style.scss の import 重複や参照漏れがないこと
- 「移動だけのつもりが rename まで混ざっていない」こと

今の計画だと、各 Step の終わりがほぼ「style.scss 更新、ビルドして確認」で閉じているため、物理統合フェーズに必要な確認が少し薄い。

提案:
- Step 2 から Step 7 には、ビルド確認とは別に「移動・統合完了チェック」を入れる
- 具体的には各 Step の最後に、次の 3 点を明示するとよい
  - 統合元ファイルは空になったか / 削除してよい状態か
  - 統合先ファイルに内容が移ったか
  - style.scss の import が新構成と一致しているか

### 6. ドキュメント更新が Step 12 にまとめられているが、style.scss の構成変更と directory rename の節目では先に更新したほうが安全なものがある

重要度: Medium

Step 12 でまとめて更新する対象は次です。

- CLAUDE.md
- scss/readme.md
- _class-rename-log.md
- progress.md

このうち、全部を最後でよいとは限らない。

特に次の 2 つは、完了後回しにしすぎると、再開時や途中レビュー時の認識ズレが起きやすい。

- _class-rename-log.md
- progress.md

計画書本文でも rename のたびに _class-rename-log.md を更新すると書いているので、Step 12 に「最終確認」とだけ置くのはよい。
ただ、progress.md は directory 再編フェーズ完了時点と rename フェーズ完了時点で 2 回以上分けて更新する想定にしたほうがよい。

提案:
- Step 7 完了時点で一度 progress.md を更新する
  - ここで「構造再編フェーズ完了 / rename 未着手」を明記
- Step 10 もしくは Step 11 完了時点でもう一度更新する
  - ここで rename 完了 / directory finalization を明記
- Step 12 は README と CLAUDE.md の最終整合フェーズとして扱う

### 2. Step 4 の .l-container 移動は、Step を分けて扱ったほうがよい

重要度: High

Step 4 には、性質の違う作業が同時に入っている。

- `_content.scss` のリネームと移動
- `.l-container` の移植
- `tailwind-base.css` の整理

このうち `.l-container` の移植だけは、単なるファイル移動ではない。

- 既存の独立ユーティリティを別ファイルへ移す
- CSS 構文から SCSS 構文へ書き換える
- 出力 CSS が同等であることを別途確認する必要がある

提案:
- Step 4 を `_content.scss` → `_layout.scss` のリネーム・移動までに限定する
- `.l-container` の移植と `tailwind-base.css` の整理は次 Step に分ける
- 以降の Step 番号は 1 つずつ後ろへずらす

この分け方なら、構造変更の確認と `.l-container` の出力同等性確認を別々に閉じられる。

### 3. Step 8 の `_layout.scss` だけ、対象範囲を明文化したほうがよい

重要度: High

Step 8 は「project/ 内の SCSS を 1 ファイルずつ接頭辞除外」となっているが、`_layout.scss` だけは例外がある。

- Step 8 で触るのは `l-main`, `l-blog__*`, `l-split__*`
- `l-container-py*` と `l-container` は Step 9 / 10 まで触らない

このルール自体は妥当。
ただ、いまはその内容が作業者の読解に委ねられている。

提案:
- Step 8 の末尾に `_layout.scss` 専用の注意書きを 1 ブロック入れる
- そこに次をそのまま書く
  - Step 8 で変更する対象
  - Step 9 / 10 まで変更しない対象
  - `_layout.scss` は Step 10 完了まで rename フェーズ未完了扱い

ここは設計の問題というより、計画書の明文化不足の問題。
1 ブロック足すだけで、実行時の迷いを減らせる。

### 4. Step 3, 5, 6 は「判断してから変更する」形を明記したほうがよい

重要度: Medium

今の Step 3, 5, 6 は、読めば意図は分かるが、判断と変更が 1 つの流れに連続して入っている。

そのままでも進められるが、実行時にぶれやすいのは「どの方針で統合するか」を文章として固定しないまま変更に入れる点。

提案:
- 各 Step で、変更に入る前に次を先に確定させる
  - 同一要素統合でいくか
  - 同ファイル共存でいくか
  - どのクラスを残し、どのクラスを統合先へ寄せるか
- その判断結果を書いてから、SCSS/PHP の変更に入る

要するに、今の案は「判断しながら実装する」流れで、提案案は「判断を先に固定してから実装する」流れにするということ。

### 5. Step 1 の調査記録は、定義記録と利用記録を分けたほうがよい

重要度: Medium

行番号の扱いは、定義側と利用側で分けたほうがよい。

提案:
- 定義記録
  - 対象クラス名
  - 定義ファイル名
  - 行番号は不要
- 利用記録
  - 対象クラス名
  - 利用ファイル名
  - 行番号は任意で保持
  - 特に PHP / JS は残してよい

理由:
- SCSS 定義側は、ファイル移動と rename で行番号がすぐ古くなる
- PHP / JS の利用側は、どこを置換するかの追跡に行番号がまだ役立つ

この分け方なら、記録の寿命を保ちつつ、置換作業の補助情報も残せる

---

## 総評

計画の方向性は通っている。

- 自作 SCSS と vendor を分離したい
- layout / component / project の FLOCSS 由来の分断をやめたい
- 接頭辞を外して、将来の module.scss 化に寄せたい

この 3 本柱は計画書の中で一貫している。

そのうえで、実行計画として見ると、いくつか詰めたほうがよい点がある。問題の中心は「何をやるか」ではなく、「どこで区切って確認するか」と「どの変更を同じフェーズで束ねるか」にある。

---

## 良い点

この計画書には、実行計画としてよい点もはっきりある。

1. directory rename を最後へ回している
- 先に project → features をやらないので、途中の切り分けがまだしやすい

2. container 系 rename を独立 Step にしている
- 一度整理済みのクラスを、通常 rename と分けて扱う意識は妥当

3. vendor を先に切り出している
- 自作 SCSS と vendor の責務分離を早い段階で進める順序はよい

4. 1 ファイルずつ進める方針が明記されている
- rename を全件一括にせず、小さく進める前提は安全側

---

## 結論

この計画は、目的と方向性の整合は取れている。
修正したほうがよいのは主に「方針」ではなく「工程の区切り方」です。

第三者レビューとしては、着手前に次の 4 点を計画書へ補強すると、実行性がかなり上がる。

1. Step 2 から Step 7 に、ビルド以外の統合完了チェックを追加する
2. Step 4 を _layout.scss 移動と .l-container 移植の 2 段に分ける
3. Step 8 と Step 9/10 の境界、特に _layout.scss の扱いを明文化する
4. progress.md の更新タイミングを最終 Step 集約ではなく、中間ゲートにも置く

この補強が入れば、計画の意図を崩さずに、実行時の迷いと事故をかなり減らせる。
