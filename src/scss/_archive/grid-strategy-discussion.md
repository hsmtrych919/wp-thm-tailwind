# グリッドシステム移行戦略 ディスカッションレポート

> 生成日: 2026-02-26
> 対象: `wp-thm/src/scss/layout/_grid.scss` のグリッドシステム移行方針
> 方法: 3案についてチーム内でアドボケイト・ディベート方式で議論

---

## 1. 議論の概要

WordPress テーマの SCSS → Tailwind CSS 移行において、グリッドシステム（`.l-row`, `.c-col__*`, `.l-grid`, `.c-grid*`）の移行戦略を検討した。

### 現状のコードベース

- **32個の PHP ファイルに 124 箇所**のグリッドクラス使用
- Bootstrap4 由来の flex ベースグリッド（負マージン + パディング方式）
- ガターは CSS 変数 `--gutter` で実現、**6段階のブレークポイントごとに vw ベースで変動**
- 12カラムグリッド（`.l-row` + `.c-col__bp--n`）とブロックグリッド（`.l-grid` + `.c-grid__bp--n`）の2系統

### 検討した3案

| 案      | 概要                                                                                                      |
| ------- | --------------------------------------------------------------------------------------------------------- |
| **A案** | 既存 SCSS グリッドをそのまま維持（移行しない）                                                            |
| **B案** | Tailwind ユーティリティに全置換。PHP テンプレートも全書き換え                                             |
| **C案** | 既存クラス名を維持し、`@layer components` 内で `@apply grid/gap` に内部再定義。PHP テンプレートは変更不要 |

### ユーザーの状況（評価に影響する重要条件）

- WordPress テーマ開発者で、**これから Tailwind を学ぶ段階**
- テンプレート変更は **AI（Claude Code）が行う**ので人間の作業負荷はほぼゼロ
- **学習目的**もあるため、Tailwind のネイティブな書き方を身につけることにも価値がある

---

## 2. Advocate-B（B案支持）の主張サマリー

**核心**: C案は「Tailwind を裏に隠した」だけで、テンプレートに旧BEM命名が残り続ける限り真の移行は完了しない。技術的負債の先送りに過ぎない。

**強い論点**:

- テンプレートに Tailwind クラスが直接出現することで、毎回の開発で Tailwind が身体に染みつく
- Tailwind のバージョンアップ時、公式 migration guide がそのまま適用できる
- `@apply` は Tailwind チーム自身が非推奨としており、将来の仕様変更リスクがある
- purge/tree-shaking の恩恵を最大化できる
- 124箇所のパターンは4種類しかなく、AI一括処理で実質リスクは低い

**改良提案（B'案）**: `gap-[calc(var(--gutter)*2)]` の冗長さを `theme.extend.gap` でカスタムユーティリティ化（`gap-x-gutter`）することで解消。

---

## 3. Advocate-C（C案支持）の主張サマリー

**核心**: グリッドはテーマ全体のインフラであり、32ファイル124箇所の同時変更はレビュー不可能な巨大 diff を生む。C案はリスクゼロで SCSS 依存を断ち切り、将来のB案移行への踏み台にもなる。

**強い論点**:

- `@layer components` の中身は Tailwind ネイティブ記法で書くので、実装者の学習機会は失われない
- Tailwind アップデート時の影響範囲が1ファイルに限定される
- PHP テンプレートに一切触れないため、テンプレート変更リスクは完全にゼロ
- vw ベースのガターが `@layer components` 内で自然に統合できる
- 混在期間がゼロで、段階的移行の最良の第一ステップになる

**改良提案（D案）**: Tailwind Plugin（`addComponents`）で定義する変種。ただし CSS 変数経由のため C案で十分と結論。

## 5. 各観点の詳細分析

### 5.1 Tailwind 学習効果

**判定: B案 ◎ > C案 ○ > A案 ✗**

B案の主張「テンプレートに Tailwind クラスが出ないと学べない」には一理ある。ユーザーは「これから Tailwind を学ぶ段階」であり、`md:col-span-6` を日常的に目にすることの学習効果は高い。

ただし C案の反論も正当: `@layer components` の中身は `@apply grid grid-cols-12` 等の Tailwind ネイティブ記法であり、実装時に Tailwind の grid/gap/responsive を学ぶ必要がある。学習機会がゼロにはならない。

**ジャッジの見解**: 学習効果に差はあるが、グリッドシステムは「インフラ層」であり、一度設定すれば日常的に触る部分ではない。Tailwind の学習本番は**コンポーネント層**（ボタン、カード、ナビ等）で行われる。グリッドの移行方式が学習全体に与える影響は、B案支持者が主張するほど大きくない。

### 5.2 保守性（Tailwind アップデート追従）

**判定: C案 ◎ > B案 ○ = A案 ○**

ここは双方の主張が異なる角度から成立している。

- B案: 公式ドキュメントがそのままリファレンスになる（利点）。ただし breaking change 時に32ファイルが影響範囲（欠点）。
- C案: `@apply` 自体の将来リスクがある（欠点）。ただし変更が1ファイルに集約される（利点）。

**ジャッジの見解**: `@apply` の非推奨リスクは Advocate-B が強調するほど深刻ではない。Tailwind v4 でも `@apply` は維持されており、廃止される兆候はない。一方、32ファイルへの影響波及は確実に発生するリスク。**変更箇所の集約（C案）の方が実務上の保守性は高い**。

### 5.3 PHP テンプレート変更リスク

**判定: A案 = C案 ◎ > B案 △**

Advocate-B は「AI が一括で行うからリスクは低い」と主張するが、Advocate-C の反論は的確:

1. 32ファイル124箇所の diff は人間がレビューしきれない
2. WordPress テーマの PHP はロジックとマークアップが混在し、文脈理解が必要
3. ユニットテストがなく、全ページ目視確認が必要

**ジャッジの見解**: 「AI が実行する」ことで人間の**作業負荷**はゼロになるが、**リスク**はゼロにならない。リスクとは「何か問題が起きる確率 × 影響度」であり、作業者が人間かAIかは確率を下げるだけで影響度は変わらない。C案のリスクゼロは本物。

### 5.4 CSS 出力効率

**判定: B案 ◎ > C案 = A案 ○**

B案が理論上最も効率的。C案は未使用カラムクラスが出力される可能性がある。

**ジャッジの見解**: しかし実質的な差は**数KB**。gzip 後は数百バイト。この差がパフォーマンスに影響するサイトは存在しない。**この観点は意思決定の要因にすべきではない**。

### 5.5 段階的移行のしやすさ

**判定: C案 ◎ > A案 ✗ > B案 △**

C案は A → C → B という段階的パスの中間地点として機能する。B案は「グリッドは一括推奨」と主張するが、これは段階的移行を放棄していることの裏返し。

**ジャッジの見解**: C案の「B案への踏み台になる」という主張は非常に重要。C案を経由することで、SCSS依存の排除とテンプレート改修を**分離**できる。問題発生時の原因切り分けが容易になる。

### 5.6 混在期間の管理

**判定: A案 = C案 ◎ > B案 △**

Advocate-B のブランチ戦略は理論上正しいが、「ブランチの作業期間中は混在状態」であることに変わりない。他の SCSS 移行作業と並行する場合、マージコンフリクトのリスクも生じる。

**ジャッジの見解**: C案の「混在ゼロ」は圧倒的な利点。CSS の内部実装だけが切り替わるため、他の移行作業との干渉もない。

### 5.7 vw ベース CSS 変数ガターの扱い

**判定: C案 ◎ > B案 △ > A案 ◎（既存のまま）**

Advocate-C の指摘は鋭い: B案でも結局 `gap-[calc(var(--gutter)*2)]` という arbitrary value を使わざるを得ない。B'案の `theme.extend.gap` で `gap-x-gutter` にできるが、これは Tailwind のデフォルトユーティリティではなくカスタム定義であり、「Tailwind ネイティブ」という B案の利点が減殺される。

**ジャッジの見解**: vw ベースの流体ガターは Tailwind の固定値 gap（`gap-4`, `gap-8`）とは根本的に相性が悪い。どちらの案でも CSS変数経由になるが、C案の `@layer components` 内でまとめて扱う方が自然で見通しが良い。

### 5.8 代替案の可能性

**B'案（theme.extend によるカスタムユーティリティ併用）**: B案の実質的な改良版。arbitrary value の冗長さを解消するが、テンプレート変更リスクは残る。

**D案（Tailwind Plugin）**: C案の変種。`addComponents` で JS 側に定義を移すが、CSS変数経由のため C案以上のメリットがない。

**ジャッジ独自の提案 — C+案（ハイブリッド段階移行）**:

> C案をベースにしつつ、**新規テンプレートからは Tailwind ユーティリティを直接使う**ルールを設ける。

具体的には:

1. **既存テンプレート**: C案で対応（クラス名維持、内部再定義）
2. **新規テンプレート・新規コンポーネント**: B案で対応（Tailwind ユーティリティ直接使用）
3. **既存テンプレートのリファクタリング時**: 触ったファイルから順次 Tailwind ユーティリティに置換

これにより:

- 既存コードのリスクゼロ（C案のメリット）
- 新規コードで Tailwind 学習（B案のメリット）
- 自然な速度で段階的にB案状態に収束
- 「いつまでに全部変える」という期限プレッシャーがない

## 6. 最終推奨案

### 推奨: C+案（C案ベース + 新規コードはB案ルール）

**理由**:

1. **リスク最小化が最優先**: 32ファイル124箇所の同時変更は、AI実行であってもレビュー負荷とテスト負荷が高い。グリッドはテーマ全体のインフラであり、ここが壊れると全ページに影響する。安全側に倒すべき。

2. **学習効果は確保できる**: C案でもグリッドの `@layer components` 実装時に Tailwind の grid/gap/responsive を学ぶ。さらに新規コードでは Tailwind ユーティリティを直接使うため、B案の学習メリットも享受できる。コンポーネント層（ボタン、カード、ナビ等）こそが Tailwind 学習の本番であり、グリッドのインフラ層だけで学習効果を論じるのは視野が狭い。

3. **技術的負債にならない**: C案は「B案への踏み台」として設計されている。将来テンプレートを触る機会に順次 Tailwind ユーティリティに置換すれば、自然にB案状態に収束する。永久にC案のまま放置されるリスクは、新規コードにB案ルールを適用することで防げる。

4. **vw ガターとの相性**: このプロジェクト固有の vw ベース流体ガターは、`@layer components` 内でまとめて扱うのが最も見通しが良い。

5. **Advocate-B の最も強い論点（`@apply` 非推奨リスク）への回答**: グリッドのような「インフラ的コンポーネント」は `@apply` の正当なユースケースとして Tailwind 公式ドキュメントでも認められている。また、将来 `@apply` が廃止される場合でも、影響は1ファイルに限定されており、そのタイミングでB案に移行すればよい。

### 純粋B案を推奨しない理由

Advocate-B の議論は論理的に整合しており、**理想論としては正しい**。しかし以下の実務的判断で C+案を優先した:

- 「AI がやるからリスクゼロ」は過信。AI の出力は人間がレビューする必要があり、124箇所の diff レビューは現実的に困難
- 一括移行のビッグバンリスクを「Git ブランチで解決」は、リスクの軽減であって排除ではない
- ユーザーは Tailwind 初学者であり、グリッドインフラの全面改修を最初のタスクにすると、問題発生時に Tailwind の問題なのか移行の問題なのか切り分けできない

### 純粋A案を推奨しない理由

- SCSS 依存を残すことは Tailwind 移行の目的に反する
- 負マージン + パディング方式は技術的にレガシーであり、CSS Grid gap に移行する価値がある

## 7. 実行計画（C+案）

### Phase 1: グリッドインフラの内部再定義（C案部分）

1. `:root` の CSS 変数定義（`--unit`, `--space`, `--gutter`, `--gutter-row`）を `@layer base` に移行
2. `@layer components` 内で以下を定義:
   - `.l-row` → `display: grid; grid-template-columns: repeat(12, minmax(0, 1fr)); gap: calc(var(--gutter) * 2);`
   - `.l-row--container` → `margin: auto; max-width` のレスポンシブ定義
   - `.c-col__bp--n` → 対応する `col-span-n`（使用されているクラスのみ）
   - `.l-grid` → `display: grid; gap: calc(var(--gutter) * 2);`
   - `.c-grid__bp--n` → 対応する `grid-cols-n`
3. 旧 `_grid.scss` の flex ベース実装を削除
4. 全ページの目視確認（ガター幅のピクセル単位検証）

### Phase 2: 新規コードのルール策定

5. チーム内ルールを文書化:
   - 新規テンプレート・新規コンポーネントでは Tailwind ユーティリティを直接使用
   - `gap-x-gutter`, `gap-y-gutter-row` を `theme.extend.gap` で定義（B'案のアイデアを採用）
   - 既存テンプレートを触る際は、可能なら Tailwind ユーティリティに置換

### Phase 3: 段階的な直接ユーティリティ化（任意・長期）

6. リファクタリングやデザイン改修の機会に、既存テンプレートの `.l-row` → `grid grid-cols-12 gap-x-gutter` 等に順次置換
7. 全テンプレートの置換完了後、`@layer components` のグリッド定義を削除

---

> **注記**: このレポートはコードの変更を伴わない。方針決定後の実装は別タスクで行う。
