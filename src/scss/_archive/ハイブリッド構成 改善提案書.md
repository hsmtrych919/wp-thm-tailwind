## 1. 現状評価サマリー

### 良い点

移行の完成度は高い。以下の点は安定しており、変更の必要がない。

- **@layer components の徹底**: 全コンポーネント・プロジェクトファイルが `@layer components` で囲まれており、Tailwind utility との優先度関係が正しく成立している
- **CSS 変数基盤の整備**: 色・gutter・header-height・form 系が `:root` / `.p-form` スコープで定義済み。Tailwind config も CSS 変数を参照しており、値の一元管理ができている
- **ビルドパイプラインの設計**: SCSS → concat → PostCSS の 3 段構成は、SCSS 関数を先に解決してから Tailwind に渡すという要件に対して正しい設計。ビルド時間も約 1.2 秒と実用的
- **namespace 統一**: 全ファイルが `@use "../global" as g;` で共通 API にアクセスする一貫したパターン
- **readme.md の完成度**: 現況ドキュメントとして十分な情報量。新規参入者でも構成を追える

### 注意が必要な点

以下は「壊れている」わけではないが、品質・保守性・Tailwind 活用の観点で改善余地がある。

## 3. A: コード品質の改善（低リスク）

### A-1: ハードコード色の CSS 変数化 — 保留

**現状**: Phase 4 で SCSS 変数 → CSS 変数の移行を行った際、`color.scale()` で計算していた派生色がハードコード化された。

**再検証結果**（2026-03-18）:
- `#47a73c`（clr1 -10%）: 4 箇所（form, button×2, tailwind-base-layer の strong）
- `#4bb140`（clr1 -5%）: 3 箇所（form, typ×2）
- `#67c45d`（clr1 +14%）: 2 箇所（post）
- 他は 1 箇所ずつ

**保留の理由**: Phase 4 で意図的にハードコード化した経緯がある。CSS 変数化すれば一括変更可能になるが、これらの色を一括で変えるユースケースが現実にあるかどうか次第。既に `--clr1-hover: #3f9536` が定義済みで `#47a73c` と近い値だが同一ではない（用途が違う）。デザイン変更時にまとめて対応する方が効率的。

### B-1: `$space_values` の二重管理解消 — 完了

**対応日**: 2026-03-18

**調査結果**: `$space_values` は build graph 内の SCSS ファイルから一切参照されていなかった。Phase 2 で utility 生成が Tailwind に移行した時点で不要になっていた。

**実施内容**:
- `global/_variables.scss` から `$space_values` マップを削除（コメントで Tailwind 移行済みの旨を記載）
- `tailwind.config.js` の spacing コメントを「旧 $space_values に基づく。SCSS 側は削除済み」に更新

### B-2: gutter 系 CSS 変数の整理可能性 — 対応しない

vw ベースのレスポンシブサイズが設計の根幹にあるため、安易な変更は表示崩れを招く。

## 6. D: 長期的な検討事項（参考）

### D-1: `@extend` の段階的解消 — 対応しない

動作しており、既存の BEM クラス構造と密接に結合している。解消するには PHP テンプレート側の変更も必要で、影響範囲が大きい。

### D-2: FontAwesome 5 → 6 移行 — 対応しない

現状動作しており、アイコンの使用箇所が SCSS mixin（`g.fa5-far()`, `g.fa-icon`）と密接に結合している。移行コストが高い。

### D-3: Sass 関数の CSS ネイティブ関数への段階置換 — 対応しない

`g.rem()` は人間が px 値で記述してビルド時に rem へ変換する意図的な設計。暗算で rem を計算する必要がなく、コードの可読性と編集効率のために維持している。`g.get_vw()` もビルド時に解決されるためランタイムコストがない。

## 7. 対応しなくてよいもの

以下は調査した結果、変更不要と判断した項目。

### breakpoints の二重定義（SCSS + Tailwind）

readme.md に記載のとおり、用途が異なる（SCSS は `@media #{g.$md}` でクラス内展開、Tailwind は `md:` で PHP テンプレート）。統一するには全コンポーネント SCSS をリライトする必要があり、現状の共存設計が正しい。

### `@layer components` 外のベンダー CSS

FontAwesome の `@font-face` は `@layer` 外が正しい配置。Swiper / Micromodal / scroll-hint の `@keyframes` も `@layer` 外で正しく配置されている。

### `tailwind.config.js` の v3 形式

v4 では CSS-first config が推奨されるが、v3 形式の JS config も `@config` ディレクティブ経由で完全にサポートされている。現状の JS config は機能しており、移行の必然性がない。

### `important` 付きの Tailwind import

`@import "tailwindcss" important;` は意図的な設計。`@layer components` 内のスタイルが Tailwind utility で上書き可能であることを保証している。

### ビルドパイプラインの 3 段構成

concat ステップを PostCSS に統合する案もあるが、現状のパイプラインは明確で保守しやすい。最適化しても 100ms 程度の短縮にしかならず、複雑さが増すデメリットの方が大きい。
